## 2. 효과적이었던 프롬프트 패턴

### 2.1 컨텍스트를 먼저 제공하기
```
❌ "슬라이드 만들어줘"
✅ "docs/ 폴더 학습하고, 배점 기준에 맞게 발표 대본 만들어줘"
```
→ 프로젝트 문서를 먼저 읽히면 엉뚱한 내용 생성이 줄어든다.

### 2.2 단계를 나눠서 요청하기
```
❌ "슬라이드 만들고 대본도 만들고 배포도 해줘"
✅ 슬라이드 구성 아이디어 요청 → 확인 → 제작 → 수정 반복
```
→ 한 번에 너무 많이 요청하면 중간 결과물을 검토하기 어렵다.

### 2.3 수정 시 "처음으로 되돌려줘" 활용
→ AI가 과도하게 단순화한 경우, 이전 버전으로 롤백 요청이 효과적이었다.

### 2.4 설명 먼저 요청 후 수정
```
"이 내용 설명해줘" → 내용 확인 → "이렇게 수정해줘"
```
→ 슬라이드 내용이 맞는지 먼저 확인하고 수정하면 실수를 줄일 수 있다.

## 3. AI가 잘 못한 것 & 직접 개입한 사례

| 상황 | AI의 한계 | 해결 방법 |

| 발표 대본 톤 | 너무 딱딱한 문어체 | 직접 읽어보며 구어체로 수정 예정 |

## 4. AGENTS.md 단일 통합 기법

### 배경
처음에는 `CLAUDE.md`가 `@AGENTS.md`를 참조하는 2파일 구조였으나,
AI가 컨텍스트를 파악하는 데 불필요한 단계가 생겼다.

### 해결책
`AGENTS.md` 하나에 아래 내용을 모두 통합:
- **Context**: 프로젝트 개요, 기술 스택
- **Rules**: 레이어 분리, 타입 규칙, 보안 규칙
- **Commands**: 자주 쓰는 CLI 명령어
- **Skills**: AI 작업 지침 (언제 무엇을 확인할지)
- **ADR**: 설계 결정 요약 (AI가 왜 이런 구조인지 이해)

### 효과
- AI가 첫 번째 메시지부터 프로젝트 구조를 정확히 파악
- 레이어 위반 코드 생성 빈도 감소
- 반복 설명 없이 일관된 결과물 생성

## 5. LLM 활용 교훈 (Lessons Learned)

1. **AI는 기억이 없다** — 매 대화마다 컨텍스트를 파일로 제공해야 한다. AGENTS.md가 그 역할을 한다.

2. **AI의 초안을 그대로 쓰지 않는다** — 슬라이드 문구, 대본 톤은 반드시 직접 검토하고 수정한다.

3. **실제 경험 데이터는 내가 제공한다** — 시행착오 사례, 수치, 맥락은 AI가 만들어낼 수 없다. 내가 알려줘야 한다.

4. **파일 기반 협업이 효과적이다** — 대화로만 작업하면 컨텍스트가 유실된다. 문서화 → AI 학습 → 작업의 사이클이 효율적이다.

5. **배점 기준을 먼저 AI에게 알려준다** — 평가 항목을 제공하면 AI가 누락 없이 커버해준다.
