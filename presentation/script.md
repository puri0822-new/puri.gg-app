안녕하세요. 저는 LOL 내전 전적 관리 웹 사이트를 모바일 앱으로 전환한 프로젝트를 발표하겠습니다.

### 【슬라이드 2】 비전 & 문제 정의 — *약 40초*
> ✅ 체계성: 비전 표현 / 비전 공감대 / 문제 정의 / 문제 공감대 / 대사 준비

"단일 코드베이스, 두 플랫폼"이라는 비전 아래 시작한 프로젝트입니다.

리그오브레전드를 직접 하지 않으시는 분들도 이런 상황은 공감하실 거라 생각합니다. **학교 풋살 동아리의 경기 결과는 FIFA나 공식 기관에 기록되지 않습니다.** 누군가 직접 기록하지 않으면 득점, 승패 모두 사라지죠. 이 프로젝트도 마찬가지입니다.

리그오브레전드에서 친구들끼리 하는 내전 경기는 Riot이 공식 데이터를 제공하지 않습니다. op.gg 같은 전적 사이트에서도 조회가 불가능하고, 저희 팀은 오랫동안 엑셀로 직접 관리해왔습니다. 그런데 경기 중에 모바일로 확인하려면 브라우저를 켜고 URL을 직접 입력해야 했습니다.

이 불편함을 해결하기 위해 **기존 웹 서비스를 모바일 앱으로 전환**했습니다.

### 【슬라이드 3】 WBS & 기술 스택 — *약 30초*
> ✅ 체계성: WBS 표현 / 적용 기술 표현
> ✅ 개발자소양: 적용 기술 설명 (읽지 않고)

총 6주의 개발 일정으로 진행했고, 현재 모든 주차가 완료된 상태입니다.

기술 스택은 네 가지입니다. 먼저 **React Native와 Expo**로 iOS와 Android를 하나의 코드베이스로 개발했습니다. 백엔드는 **Firebase Firestore**를 사용해 실시간 데이터 구독을 구현했고, **NativeWind**로 기존 웹의 Tailwind CSS 스타일을 그대로 재사용했습니다. 배포는 **EAS Build와 GitHub Actions**로 자동화했습니다.

### 【슬라이드 4】 아키텍처 & 앱 구조 — *약 30초*
> ✅ 개발자소양: 아키텍처 이해하고 설명
> ✅ 질의응답: 앱 구조 (레이어 + 디렉토리)

앱은 세 개의 레이어로 구성되어 있습니다.

**화면 레이어**는 `app/` 디렉토리로, UI 렌더링만 담당합니다. **데이터 레이어**는 `src/hooks/`로, Firestore의 `onSnapshot`을 통해 실시간 구독을 처리합니다. **비즈니스 로직 레이어**는 `src/utils/`로, ELO와 CP 계산 함수들이 React나 Firebase에 의존하지 않는 순수 TypeScript 함수로 구성되어 있습니다.

외부로는 Firebase Firestore와 Riot Data Dragon CDN을 연결해 사용합니다.

---

### 【슬라이드 5】 진행 과정 & 시행착오 — *약 35초*
> ✅ 체계성: 프로젝트 진행 및 완료 / 어떻게 구현했는지
> ✅ 개발자소양: 구현 시행착오 (개발자 관점)

개발 과정에서 세 가지 주요 시행착오가 있었습니다.

첫 번째는 **웹과 앱의 ELO 불일치** 문제입니다. 웹사이트의 ELO 계산 공식을 변경했는데, 앱은 구버전 로직을 그대로 유지하고 있어서 동일한 데이터임에도 수치가 달랐습니다. 이를 `elo.ts` 하나를 단일 소스로 통합해 웹과 앱이 동일한 로직을 공유하도록 해결했습니다.

두 번째는 **CP 불일치**로, 플레이어 통계 페이지 개발 중 같은 방식으로 해결했습니다.

세 번째는 **전적 페이지 과부하**입니다. 모든 경기 기록을 한 번에 불러오다 보니 로딩이 느려졌고, 20개씩 커서 페이지네이션을 적용해 해결했습니다.

### 【슬라이드 6】 성능 최적화 · 코드 품질 · 테스트 — *약 30초*
> ✅ 개발자소양: 성능 최적화 / 코드 품질 / 단위 테스트 / 통합 테스트 / 개선 의지

성능 최적화로는 FlatList에 `keyExtractor`와 `getItemLayout`을 적용해 스크롤 성능을 개선했고, 검색에 300ms 디바운스를 적용해 불필요한 Firestore 요청을 줄였습니다.

코드 품질은 TypeScript strict 모드와 ESLint, Prettier를 통해 관리했으며, 레이어 분리 원칙을 엄격하게 지켜 화면에서 비즈니스 로직이 직접 실행되지 않도록 했습니다.

테스트는 소환사 검색, 리더보드, 전적 더보기, 통계 정렬, CP·MVP 배지, Pull-to-Refresh 등 핵심 기능을 수동으로 검증했습니다.

배포는 `git tag v*.*.*`를 푸시하면 GitHub Actions가 자동으로 EAS Build를 실행해 APK를 생성하고, GitHub Releases에 업로드합니다. 지인들은 해당 링크에서 APK를 다운로드해 설치하면 됩니다.

빌드 프로필은 debug, preview, release 세 단계로 나뉘며, release 빌드만 프로덕션 환경과 Keystore 서명이 적용됩니다.

### 【슬라이드 10】 ADR 요약 (질의응답 대비) — *슬라이드만 표시, 질문 시 답변*
> ✅ 질의응답: ADR 3개 이상 / 정확한 답변

> **예상 질문 & 답변 준비**
>
> **Q. 왜 TanStack Query를 사용하지 않았나요?**
> A. Firebase Firestore는 `onSnapshot`으로 실시간 구독을 네이티브 지원합니다. TanStack Query는 REST API 기반 캐시 관리에 적합한데, 이미 실시간 스트림이 제공되는 환경에서는 추가 캐시 레이어가 불필요하다고 판단했습니다. (ADR-002)
>
> **Q. 왜 Flutter가 아닌 Expo를 선택했나요?**
> A. 기존 웹 서비스가 TypeScript와 React로 작성되어 있어, `elo.ts`와 `cp.ts` 같은 핵심 비즈니스 로직을 수정 없이 그대로 이식할 수 있었기 때문입니다. (ADR-004)
>
> **Q. 왜 앱에서 쓰기 기능을 제외했나요?**
> A. 클라이언트에서 데이터를 직접 수정하면 보안 규칙이 복잡해지고 변조 위험이 생깁니다. 읽기 전용으로 제한하면 Firestore Security Rules가 단순해지고 데이터 무결성을 보장할 수 있습니다. (ADR-001)
